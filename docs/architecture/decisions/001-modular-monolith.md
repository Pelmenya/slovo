# ADR-001: Modular Monolith вместо микросервисов

## Статус
✅ Принято — 2026-04-22

## Контекст

Проект начинается как pet-проект с прицелом на SaaS. Нужно решить:

- Строить монолит — проще на старте, но "грязные" границы потом рвутся с болью
- Строить микросервисы — чистые границы с первого дня, но 10x инфраструктурной сложности

У проекта:
- Один разработчик
- Ограниченное время (вечера, выходные)
- Неизвестный продуктовый фит — фичи могут меняться радикально
- Требования к observability, rate limiting, мониторингу

## Решение

**Modular Monolith** — единый деплоймент с чёткими модульными границами:

- `apps/api` — HTTP-фронтенд
- `apps/worker` — RabbitMQ consumer
- `libs/<feature>` — изолированные фичи

Один `docker-compose up` — весь стек работает. Внутри — структура, которая легко разрезается на микросервисы когда будет нужно.

## Альтернативы

### Микросервисы с первого дня

Плюсы:
- Чистые границы с самого начала
- Разные сервисы можно писать на разных языках
- Независимое масштабирование

Минусы:
- **Service discovery** (Consul / etcd)
- **Distributed tracing** (OpenTelemetry) обязателен — без него debugging ад
- **API Gateway** нужен
- **Inter-service auth** (mTLS / JWT)
- **Eventual consistency** между сервисами — сложность кода растёт
- Локальная разработка требует 10+ контейнеров

Для pet-проекта соло-разработчика это **стартовать не даст**. Проект умрёт в инфраструктуре.

### Serverless (AWS Lambda / Vercel Functions)

Плюсы:
- Автомасштабирование
- Почасовая оплата

Минусы:
- Холодный старт (cold start) = плохой UX для первого запроса
- LLM-запросы 10-30 секунд = дорого в serverless
- Stateful части (WebSocket, long polling) не подходят
- Vendor lock-in

### Чистый монолит без модулей

Минусы:
- Через 6 месяцев → "big ball of mud"
- Фичи переплетаются, тесты хрупкие
- Не масштабируется организационно (если появится второй разработчик)

## Последствия

### Плюсы

- ✅ Быстрый старт — `npm run start:dev` и всё работает
- ✅ Простой деплой — один Docker образ, одна команда
- ✅ Легко рефакторить — всё в одной кодовой базе
- ✅ Легко тестировать — unit + e2e без network
- ✅ Модули уже изолированы — экстракция в микросервис = копипаст + HTTP-клиент
- ✅ Дешевле хостить (1 сервер vs cluster)

### Минусы

- ⚠️ Единая точка отказа — упала БД = упало всё
- ⚠️ Невозможно масштабировать одну фичу независимо
- ⚠️ Единый язык (TypeScript) — нельзя взять Python для ML-части

### Когда пересмотреть

- Любая фича требует горизонтального масштабирования которое монолит не потянет
- Появилась команда 5+ разработчиков, конфликты в коде
- Нужен другой язык (Python для тяжёлого ML)
- Фича требует изоляции для compliance (PCI/HIPAA)

В каждом из этих случаев **конкретный модуль** выносится в микросервис. Остальной монолит остаётся монолитом.
