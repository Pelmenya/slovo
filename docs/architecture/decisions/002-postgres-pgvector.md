# ADR-002: PostgreSQL + pgvector вместо Pinecone / Qdrant / Chroma

## Статус
✅ Принято — 2026-04-22

## Контекст

Для RAG-фич (Retrieval Augmented Generation) нужно хранилище векторов-embeddings:

- Индексация документов / товаров / записей
- Быстрый поиск по семантической близости
- Масштабирование от тысяч до миллионов записей

Варианты:

1. **Dedicated vector DB** (Pinecone, Qdrant, Chroma, Weaviate, Milvus)
2. **PostgreSQL + pgvector**
3. **Redis/Valkey + RedisSearch**
4. **MongoDB Atlas Vector Search**

## Решение

**PostgreSQL 18 + pgvector 0.8.2** — одна БД для реляционных данных и векторов.

> Версия обновлена с PG17 → PG18.3 (2026-04-22) — PG18 стал GA и является актуальным major. pgvector 0.8.2 поддерживает оба мажора, используем образ `pgvector/pgvector:0.8.2-pg18-trixie`. Бонус: на PG18 `gen_random_uuid()` доступен нативно без `pgcrypto`.

## Альтернативы

### Pinecone (managed)

Плюсы:
- Managed-сервис, нулевая инфраструктура
- Быстрый масштаб "из коробки"

Минусы:
- ❌ **Vendor lock-in** — нельзя перенести
- ❌ **Дорого** на масштабе (~$70/мес за базовый план)
- ❌ **Отдельная БД** — синхронизация с основной через код
- ❌ **Не self-hosted** — данные уходят наружу

### Qdrant (self-hosted)

Плюсы:
- Open-source, Rust, быстрый
- Богатый API

Минусы:
- ⚠️ **+1 сервис** в инфре
- ⚠️ Синхронизация с основной БД
- ⚠️ Свой язык фильтров — не SQL

### Chroma

Плюсы:
- Простой, Python-native
- Встраиваемый

Минусы:
- ⚠️ Молодой проект, API меняется
- ⚠️ В production — ограниченная поддержка
- ⚠️ +1 сервис

### MongoDB Atlas Vector Search

Плюсы:
- Интегрирован в Atlas

Минусы:
- ❌ **Только Atlas (cloud)** — нет self-hosted
- ❌ Vendor lock-in к MongoDB Atlas

### Valkey / RedisSearch

Плюсы:
- Уже есть в стеке (как кэш)
- Быстрый

Минусы:
- ⚠️ Хранение векторов в памяти — дорого на миллионах записей
- ⚠️ Нет persistence-гарантий под большие объёмы

## Последствия

### Плюсы

- ✅ **Одна БД** — реляционные данные + векторы в одной транзакции
- ✅ **SQL** — все возможности JOIN, GROUP BY, WHERE + векторный поиск
- ✅ **HNSW индекс** — 95-99% точность, быстрый
- ✅ **Бесплатно** — никаких managed-платежей
- ✅ **Self-hosted** — данные под контролем
- ✅ **Простая миграция данных** — стандартный pg_dump
- ✅ **Familiar** — пользователи Postgres знают инструмент
- ✅ **PostgreSQL 18 нативно поддерживает расширения** через Prisma 7

### Минусы

- ⚠️ **Один инстанс БД** — vector + OLTP нагрузка конкурируют за ресурсы
- ⚠️ **Ограничение по объёму** — при 10M+ векторов Pinecone/Qdrant могут быть быстрее
- ⚠️ **Prisma не поддерживает `vector` тип нативно** — см. ADR-005

### Когда пересмотреть

- Объём превысит 10M векторов и Postgres начнёт тормозить
- Требуется специфичная фича vector DB (реrerankers из коробки, hybrid search)
- Появится выделенный ML-data team, для которых Qdrant/Milvus — стандарт

В этом случае **выделяем `vector`-поля в отдельный Qdrant**, основные данные остаются в Postgres.

### Стратегия на рост

```
0 - 100k векторов        → pgvector без индексов
100k - 1M векторов       → pgvector + HNSW индекс
1M - 10M векторов        → pgvector + HNSW + раздельные таблицы по тенантам
10M+ векторов            → Переход на Qdrant (один модуль, остальное в Postgres)
```
